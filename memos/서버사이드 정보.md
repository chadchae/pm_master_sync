서버사이드(FastAPI) + JSON 파일 DB에 저장되는 모든 정보입니다.

  ---
  저장 데이터 전체 구조

  1. 시스템 전역 (backend/data/*.json)

  ┌─────────────────┬─────────────────────────────────────────────────────────────┐
  │      파일       │                            내용                             │
  ├─────────────────┼─────────────────────────────────────────────────────────────┤
  │ auth.json       │ 비밀번호 해시 (bcrypt)                                      │
  ├─────────────────┼─────────────────────────────────────────────────────────────┤
  │ tokens.json     │ 세션 토큰                                                   │
  ├─────────────────┼─────────────────────────────────────────────────────────────┤
  │ people.json     │ 협업자/인물 DB (이름, 소속, 전문성, 관계, 연결관계, 중요도) │
  ├─────────────────┼─────────────────────────────────────────────────────────────┤
  │ card_order.json │ 칸반 보드 카드 순서 (stage별 프로젝트 정렬)                 │
  ├─────────────────┼─────────────────────────────────────────────────────────────┤
  │ plans.json      │ 플랜 목록 (제목, 단계, 관련프로젝트/인물/할일)              │
  └─────────────────┴─────────────────────────────────────────────────────────────┘

  ---
  2. 프로젝트별 데이터 (각 {project-slug}.json)

  todos/{project}.json
  - 할일 항목: 제목, 컬럼(todo/in_progress/done/waiting/archive), 우선순위, 담당자, 마감일, starred 여부

  issues/{project}.json
  - 이슈 트래커: 제목, 설명, 상태(open/closed), 우선순위, 라벨, 댓글 스레드

  schedules/{project}.json
  - 일정/간트: 태스크 제목, 시작일/종료일, 담당자, 상태, 의존관계, 진행률

  subtasks/{project}.json
  - 서브태스크: 제목, 상태(done/pending), 완료일

  logs/{project}.json
  - 활동 로그: create/update/milestone 이벤트, 타임스탬프, 태그

  ---
  3. 프로젝트 메타 (파일시스템 기반, ~/Projects/{stage}/{slug}/)

  서버가 직접 파일을 읽어 제공하는 것들:
  - project.yaml — 프로젝트 메타데이터 (유형, 협업여부, 단계 등)
  - docs/*.md — 문서 파일 (읽기/쓰기)
  - memo.md — 프로젝트 메모

  ---
  요약

  JSON DB (backend/data/)
  ├── 전역: 인증, 세션, 인물, 카드순서, 플랜
  └── 프로젝트별: 할일, 이슈, 일정, 서브태스크, 활동로그

  파일시스템 (~/Projects/)
  └── 프로젝트별: project.yaml, 문서(md), 메모

  현재 DB 방식: SQLite/PostgreSQL 없이 JSON 파일로 관리 — 단순하고 이식성 좋지만, 동시 접근 시 race condition 주의 필요.
